portal v0.4.1 makes the v0.4.0 credential-sharing shims **just work in every
bash shell shape** a coding agent or script might run in — no manual PATH or
`SUDO_ASKPASS` setup, no shell-startup arcana. It also hardens the shim
backup/restore path across upgrades. Dev-box shim version: **v7**.

## Fixed: bash startup coverage

v0.4.0 wrote its PATH and `SUDO_ASKPASS` blocks to `~/.bashrc`, `~/.zshrc`,
`~/.zshenv`, and `~/.profile`. zsh was fully covered, but bash had two gaps:

- **`~/.bash_profile` shadowing** — bash login shells read the first of
  `.bash_profile`/`.bash_login`/`.profile` *only*, so on boxes with a
  `.bash_profile` the blocks in `.profile` never ran for `bash -l` / `bash -lc`
  (a common agent-harness shell shape). Install now appends the blocks to an
  **existing** `.bash_profile`/`.bash_login` too. It never *creates* those
  files — that would shadow your `.profile`.
- **Non-interactive `.bashrc` guard** — Debian/Ubuntu bash sources `.bashrc`
  even for ssh one-shots (`ssh box 'cmd'`), but the distro's interactive guard
  returns before reaching blocks appended at the bottom. Install now also
  places a small **early PATH block at the top** of `.bashrc`, above the guard,
  so non-interactive bash gets the shims; the bottom block still re-wins over
  nvm/conda in interactive shells.

Net effect: `ssh <box> 'sudo systemctl restart foo'` from your Mac now pops the
native approval dialog instead of dying with "a terminal is required".

## Improved: `sudo` shim self-selects its askpass helper

A detached `sudo` no longer depends on any rc file having exported
`SUDO_ASKPASS`: when it is unset, the shim selects `~/.local/bin/portal-askpass`
itself (a helper you configured yourself is never overridden). Scenario 2 now
needs only PATH. The human safety invariant is unchanged — any session with a
reachable controlling terminal is a byte-for-byte passthrough to real sudo.

## Fixed: upgrade-safe shim backups

Shim backup/restore now recognizes portal's own shims of **any** version.
Previously, upgrading over an older shim with an empty backup slot copied the
outgoing shim into the backup, and a later `portal uninstall` would "restore"
that stale shim instead of deleting it. Upgrades no longer pollute backup
slots, and uninstall deletes portal-owned backups it finds (healing boxes
already affected) while still restoring a genuine pre-existing binary.

## Remaining limit (unfixable by design)

A plain non-interactive `sh -c`/dash process from a clean environment sources
no startup file at all — POSIX gives no hook. Such processes inherit only
their parent's environment, so they reach the shims only if that parent's PATH
includes `~/.local/bin`. Every shell an agent realistically runs in
(interactive, login, ssh one-shot bash, any zsh) is covered.

## Install / upgrade

```sh
curl -fL -o portal-darwin-arm64 \
  https://github.com/VikashLoomba/Portal/releases/latest/download/portal-darwin-arm64
chmod +x portal-darwin-arm64
./portal-darwin-arm64 install <ssh-host>
```

Upgrading from v0.4.0: run the same `install` — it replaces the Mac binary,
redeploys the v7 shims, converges the new shell-startup blocks, and re-runs
the `portal doctor` self-test. No protocol change. Apple Silicon (arm64) Macs
only.

portal v0.5.0 makes portal **embeddable**: a desktop app can now bundle the
portal binary as a sidecar and drive everything — including first-run remote
setup — through the local API, so end users install only the app, never
portal itself.

## New: `POST /v1/setup` — remote bootstrap over the local API

Setup is no longer CLI-only. The daemon exposes the full remote bootstrap
(validate → configure → deploy the agent → activate) as a streaming ndjson
endpoint the TypeScript SDK wraps as `setup()`:

```ts
for await (const event of setup(sock, { host: "me@devbox" })) {
  // running / line / warning / failure / done — render as onboarding UI
}
```

Step failures arrive in-band, ssh's own stderr is relayed as `line` events,
and `force: true` replaces the CLI's `[y/N]` reconfigure prompt.

## New: unconfigured idle mode

`portal run` with no configured host now starts and serves the local API
instead of exiting. An embedding app can ship a zero-config sidecar, read
`status.host === ""`, and show onboarding — the daemon is already alive and
waiting for `/v1/setup`.

## New: in-process hot-swap activation

Completing setup activates the new host inside the running daemon: the new
stack is constructed first, the old one drains fully, and the already-open
`/v1/events` stream simply starts reporting the configured host. No daemon
restart, no socket change, no PID change — and a failed setup never degrades
existing connectivity. Setup uses a dedicated ssh ControlPath, so bootstrap
traffic can never ride (or hijack) the live connection's ControlMaster during
a host switch.

## Changed: the reference embedding is now Deno Desktop + TanStack Start

`examples/shell-electron` is replaced by `examples/shell-desktop`, built on
Deno 2.9's experimental `deno desktop` runtime. It demonstrates the full
embedding recipe: app-scoped sidecar supervision (`PORTAL_CONFIG_DIR`,
`PORTAL_API_SOCK`, `PORTAL_SOCK`), streamed first-run setup as onboarding UI,
an events-driven shell, and interactive PTY exec (vim works) through a
capability-gated WebSocket bridge. The unmodified SDK runs under Deno's node
compatibility layer. `docs/embedding.md` documents the recipe, including why
`PORTAL_SOCK` must always be set explicitly for an embedded sidecar.

## Notes

- The TypeScript SDK gains `setup()` and `waitReady()`; wire behavior is
  unchanged (ProtoVersion stays 4) and existing daemons are unaffected.
- Persistence (launch at login) is deliberately the embedding app's job —
  e.g. SMAppService/openAtLogin — not the sidecar's.
- Deno Desktop is experimental; the example pins Deno 2.9.x and documents a
  manual validation checklist (packaged-mode exec among them) in its README.

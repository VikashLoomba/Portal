portal v0.4.0 adds **credential sharing** — a coding agent on the dev box can
ask *you* for a secret it must never see. The agent runs a command; the Mac
pops a native secure-input dialog; the secret is delivered straight into the
target process (or into `sudo`) without ever entering the agent's context,
transcript, argv, logs, or the box's disk.

It rides the same authenticated connection portal already maintains — the same
rails as clipboard paste — so there's nothing new to run and no new port.

## New: `portal keychain` — request a secret without exposing it

When an agent needs a login, it wraps the command on the **dev box**:

```sh
# The Mac shows a dialog; on approval $PW is set only in the child's environment.
portal keychain run --label "staging admin" --env PW -- \
    sh -c 'curl -sf -d "user=admin" -d "pass=$PW" http://localhost:3000/login'

# Or feed the secret on the child's stdin (e.g. docker login --password-stdin):
portal keychain run --label "registry token" --stdin -- docker login -u me --password-stdin ghcr.io
```

The single quotes matter: they make the *child* shell expand `$PW`, so the
agent's own shell never sees it. The agent only ever observes the child's
stdout/stderr and exit code. On denial or timeout the command exits non-zero
with an agent-legible message. `portal keychain --help` is written for an agent
to read and figure out on its own.

## New: transparent `sudo`

When an agent (with no controlling terminal) runs `sudo <cmd>`, the Mac pops the
same dialog and the password flows straight into `sudo`'s askpass pipe:

```sh
sudo apt install -y some-package    # Mac dialog → command proceeds
```

Interactive `sudo` at a real terminal is **untouched** — portal only steps in
for a fully detached (agent) invocation, and never replaces a human's own
password prompt.

## Remember with the macOS Keychain

The first approval dialog offers **Allow & Remember**. Choosing it stores the
secret in the macOS Keychain (`portal-cred` service); later requests for the
same label become a one-click **Allow** confirmation — no retyping. Manage
remembered credentials on the Mac:

```sh
portal keychain list            # remembered labels
portal keychain forget <label>  # drop one
```

## Safety and scope

- **Feature gate:** on by default (nothing is served without a per-request
  click); disable entirely with `portal features cred off`.
- **Audit:** every approval and denial is recorded (label, requester, mode,
  decision, duration) in the append-only audit log — the secret never is.
- **Threat model:** the guarantee is that the secret never reaches the agent's
  context, argv (either machine), portal logs, or the box's disk. It is **not**
  a defense against a malicious process running as the same box user (which can
  read `/proc/<pid>/environ`). The consent dialog and audit log are the control
  points. Transparent `sudo` fires only for an agent with no controlling
  terminal; in a shared interactive SSH session use `portal keychain run` or
  approve the `sudo` yourself.

## Install / upgrade

```sh
curl -fL -o portal-darwin-arm64 \
  https://github.com/VikashLoomba/Portal/releases/latest/download/portal-darwin-arm64
chmod +x portal-darwin-arm64
./portal-darwin-arm64 install <ssh-host>
```

Upgrading from v0.3.0: run the same `install` — it replaces the binary in
`~/.local/bin`, redeploys the dev-box agent and shims (adding the `portal`,
`portal-askpass`, and `sudo` credential shims and the `SUDO_ASKPASS` export),
and re-runs the `portal doctor` self-test. No protocol change — a v0.3.0 Mac and
a v0.4.0 box interoperate; credential requests simply need both sides on v0.4.0.
Apple Silicon (arm64) Macs only.
